# 双库推送指南（Gitee 私有 + GitHub 公共）

本项目同时维护两个远端仓库：

- Gitee（私有仓库）：https://gitee.com/SeniorAgentTeam/jin-ce-zhi-suan.git
- GitHub（公共仓库）：https://github.com/Scottzt/jin-ce-zhi-suan.git

目标是：

- Gitee `main`：包含桌面端打包/托盘等私有仓库需要的内容
- GitHub `main`：只包含可公开的核心代码（不包含桌面端打包/托盘相关文件）

由于 `.gitignore` 是“随分支提交”的文件，Git 无法对同一分支按不同 remote 应用不同 `.gitignore`。因此采用“两分支 + 两远端”的做法。

---

## 1. 远端命名约定

本地建议保持以下 remote：

- `private` → Gitee
- `origin`（或 `public`）→ GitHub

查看/设置远端：

```bash
git remote -v

git remote add private https://gitee.com/SeniorAgentTeam/jin-ce-zhi-suan.git
git remote set-url origin https://github.com/Scottzt/jin-ce-zhi-suan.git
```

---

## 2. 分支约定

- `main`：私有分支（推送到 Gitee），包含桌面端相关代码
- `public-main`：公开分支（推送到 GitHub），不包含桌面端相关代码

创建公开分支（只需一次）：

```bash
git fetch origin
git checkout -B public-main origin/main
```

---

## 3. 推送策略

### 3.1 推送到 Gitee（私有）

在 `main` 分支开发与提交，然后推送到 Gitee：

```bash
git checkout main
git push private main
```

### 3.2 推送到 GitHub（公开）

公开仓库采用 `public-main` 分支。推荐用 cherry-pick 将“可公开的提交”同步过去：

```bash
git checkout public-main
git cherry-pick <commit_sha>
git push origin public-main:main
```

说明：

- 如果某个提交包含桌面端/打包相关文件，请不要 cherry-pick 到 `public-main`
- `dist/`、`build/` 等产物不建议进入 Git 历史，推荐通过 Release/附件发布

---

## 4. 为两个远端配置默认推送目标（可选，但强烈推荐）

配置后：

- `git push private` 会把本地 `main` 推到 `private/main`
- `git push origin` 会把本地 `public-main` 推到 `origin/main`

```bash
git config remote.private.push refs/heads/main:refs/heads/main
git config remote.origin.push refs/heads/public-main:refs/heads/main
```

查看配置：

```bash
git config --get-all remote.private.push
git config --get-all remote.origin.push
```

---

## 5. GitHub（HTTPS）鉴权方式（PAT）

GitHub 远端是 HTTPS 时，建议使用 Personal Access Token（PAT）作为密码。

步骤：

1) GitHub 生成 PAT（Settings → Developer settings → Personal access tokens）
2) 推送时用户名填 GitHub 用户名，密码填 PAT

如果之前缓存了错误凭据（macOS 钥匙串），可清理后再推送：

```bash
git credential-osxkeychain erase <<EOF
protocol=https
host=github.com
EOF
```

---

## 6. 约束与建议

- `.gitignore` 差异：请只在 `public-main` 分支中增加“公开仓库不应出现的文件”忽略规则
- `main` 分支可以保留桌面端相关文件（例如 `desktop_launcher.py`、`desktop.spec`、`scripts/build_desktop.*`）
- `public-main` 分支不应包含任何密钥/私有配置/桌面端打包脚本

---

## 7. Windows + macOS 双环境开发：提交与推送（详细）

你可以在 Windows 和 macOS 上同时开发，但建议所有改动都遵循同一套提交规则，避免把“桌面端/构建”意外推到 GitHub。

### 7.1 提交前：更新与检查改动

每次开始开发前（无论在哪台机器）：

```bash
git checkout main
git pull private main
```

开发完成后，提交前先检查：

```bash
git status
git diff
```

### 7.2 拆分提交：可公开 vs 仅私有（强烈推荐）

建议把一次开发拆成两类 commit：

- 可公开提交：将来要同步到 GitHub（只改核心代码）
- 仅私有提交：只留在 Gitee（桌面端构建/托盘/平台适配）

#### A) 可公开提交（可同步到 GitHub）

特点：只改 `src/`、`server.py`、`static/` 等，不包含桌面端打包脚本与私有配置。

```bash
git add server.py src/ static/
git status
git commit -m "fix: xxx"
```

#### B) 仅私有提交（桌面端/构建/平台适配）

特点：只在 `main` 分支提交并推到 Gitee，不同步到 `public-main`。

```bash
git add desktop_launcher.py desktop.spec scripts/build_desktop.sh scripts/build_desktop.bat
git status
git commit -m "desktop(mac): xxx"
```

提示：

- 不建议 `git add .`，优先用“挑选性 add”来保证提交干净
- `dist/`、`build/` 等产物不要提交到 Git；推荐通过 Release/附件发布

### 7.3 推送到 Gitee（私有）

在 `main` 分支提交完成后：

```bash
git checkout main
git push private main
```

### 7.4 同步到 GitHub（公开）

公开仓库使用 `public-main` 分支维护。推荐用 cherry-pick 同步“可公开提交”：

1) 切换到公开分支并更新：

```bash
git checkout public-main
git pull origin main
```

2) 查看哪些提交可同步：

```bash
git log --oneline public-main..main
```

3) 将可公开提交同步到公开分支：

```bash
git cherry-pick <sha1> <sha2>
```

4) 推送到 GitHub：

```bash
git push origin public-main:main
```

### 7.5 多机同步建议（Windows ↔ macOS）

在另一台机器开始开发前都执行：

```bash
git checkout main
git pull private main
```

如果你也需要在另一台机器维护公开分支：

```bash
git checkout public-main
git pull origin main
```

### 7.6 常见问题

- 不小心把桌面端文件混进了“可公开提交”
  - 未提交：`git restore --staged <file>`
  - 已提交：建议 `git reset HEAD~1` 拆分后重新 add/commit
- GitHub HTTPS 鉴权失败：使用 PAT，并按第 5 节清理凭据缓存后重试
