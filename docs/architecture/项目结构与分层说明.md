# 项目结构与分层说明

本文说明本项目的目录分层、核心业务链路、分包职责和包之间的依赖关系。阅读目标是：知道代码应该放在哪一层、一个请求或一次回测会经过哪些模块、扩展数据源/策略/页面接口时应遵守哪些边界。

## 总体分层

项目采用“入口层 + 应用编排层 + 领域职能层 + 基础设施层 + 扩展子系统”的结构。

```text
server.py / run_backtest.py / run_live.py / main.py
        |
        v
src/core/                 应用编排层：回测、实盘、三省调度
        |
        v
src/ministries/           领域职能层：六部职能，资金、风控、执行、报告、数据清洗
        |
        v
src/strategies/           策略层：策略基类、内置策略、策略仓库、策略工厂
        |
        v
src/utils/                基础设施层：配置、数据源、指标、同步、通知、运行参数
```

外围扩展能力包括：

- `src/evolution/`：策略进化与多 Agent 策略生成评估。
- `src/consistency/`：实盘/回测一致性快照、回放、对比、报告。
- `src/strategy_intent/`：自然语言或市场状态到策略意图的解析。
- `src/tdx/`：通达信公式、终端桥接、券商网关骨架。
- `scripts/`：运维、批量任务、数据同步辅助脚本。
- `docs/`：使用说明、设计说明、表结构和操作指南。

## 顶层入口

### `server.py`

主 Web 服务入口，基于 FastAPI 暴露控制台页面和 API。

主要职责：

- 启动前依赖自检。
- 创建 FastAPI `app`。
- 提供配置中心、策略管理、回测控制、实盘控制、历史同步、策略进化、报告、一致性等 API。
- 作为前端 `dashboard.html` 的服务端。
- 调用 `BacktestCabinet`、`LiveCabinet`、`HistoryDiffSyncService`、`EvolutionRuntimeManager` 等应用服务。

`server.py` 是当前最大的集成层，不应把可复用业务逻辑继续堆在这里。新增复杂逻辑时，优先下沉到 `src/core/`、`src/utils/`、`src/evolution/` 或 `src/consistency/`，API 层只做参数接收、校验和调度。

### `run_backtest.py`

命令行回测入口。

主要职责：

- 解析 `--stock`、`--start`、`--end`、`--strategy` 等参数。
- 应用全局回测基线 `apply_backtest_baseline()`。
- 创建 `BacktestCabinet`。
- 输出回测进度、交易事件和报告摘要。
- 保存单次回测报告到 `data/reports/`。

### `run_live.py`

命令行实盘监控入口。

主要职责：

- 读取 `system.enable_live`。
- 按优先级选择数据源：环境变量 `DATA_PROVIDER` > `config.json` 的 `data_provider.source` > 默认值。
- 创建 `LiveCabinet` 并进入实盘循环。

### `main.py`

早期或演示型回测入口，直接串起三省六部对象并逐 bar 运行。当前更完整的回测能力主要在 `BacktestCabinet` 和 `run_backtest.py`。

建议：

- 新功能优先接入 `BacktestCabinet` 或 `server.py` 的 API。
- `main.py` 可作为理解三省六部调用顺序的简化参考。

### `desktop_launcher.py`

桌面启动器和打包场景入口，负责桌面环境下启动服务、打开浏览器或托盘相关能力。

## `src/core`：应用编排层

`src/core` 负责把数据源、策略、风控、执行、报告这些领域对象组织成完整流程。它不是底层工具层，也不是单个策略逻辑层。

### `BacktestCabinet`

文件：`src/core/backtest_cabinet.py`

回测主编排器。

职责：

- 根据 `data_provider.source` 构建数据源。
- 检查数据源连通性。
- 拉取不同周期 K 线。
- 初始化策略集合和每个策略的资金账户。
- 按时间顺序驱动策略生成信号。
- 调用门下省做风控审核。
- 调用尚书省执行订单。
- 调用礼部生成报告和排行。
- 通过事件回调向 WebSocket 或 CLI 输出进度。

依赖关系：

```text
BacktestCabinet
  -> ConfigLoader
  -> DataProvider / TushareProvider / AkshareProvider / MysqlProvider / PostgresProvider / DuckDbProvider / TdxProvider
  -> strategy_factory.create_strategies()
  -> ZhongshuSheng / MenxiaSheng / ShangshuSheng
  -> LiBuPersonnel / HuBuRevenue / LiBuRites / BingBuWar / XingBuJustice / GongBuWorks
```

### `LiveCabinet`

文件：`src/core/live_cabinet.py`

实盘监控主编排器。

职责：

- 选择并初始化实时数据源。
- 维护实时 K 线缓存、策略状态、持仓和资金状态。
- 周期性拉取最新行情。
- 复用三省六部流程生成信号、风控审核、执行和推送事件。
- 采集一致性快照。
- 可选同步实时数据到默认 API。

### 三省对象

三省是决策主链路：

- `CrownPrince`：太子院，数据前置校验和分发入口。
- `ZhongshuSheng`：中书省，调用策略生成信号，并同步策略持仓状态。
- `MenxiaSheng`：门下省，风控审核，决定信号是否放行。
- `ShangshuSheng`：尚书省，执行订单、维护持仓、处理止盈止损。

典型调用顺序：

```text
行情 K 线
  -> CrownPrince 数据校验
  -> ZhongshuSheng.generate_signals()
  -> MenxiaSheng.check_signal()
  -> ShangshuSheng.execute_order()
  -> HuBuRevenue / XingBuJustice / LiBuRites 更新资金、风险和报告
```

## `src/ministries`：六部领域职能

六部是领域服务层，每个包专注一个业务能力。

### `li_bu_personnel.py`

吏部：策略注册与策略名单管理。

职责：

- 注册策略对象。
- 提供策略集合管理能力。

### `hu_bu_revenue.py`

户部：资金、现金、净值、成本核算。

职责：

- 维护现金和账户价值。
- 记录交易流水。
- 计算资金曲线和收益指标所需数据。

### `li_bu_rites.py`

礼部：报告、统计、排行榜。

职责：

- 从策略、资金、风控记录中生成绩效报告。
- 计算 ROI、年化、最大回撤、胜率、盈亏比、Sharpe、Calmar 等指标。
- 生成策略排行。

### `bing_bu_war.py`

兵部：执行撮合。

职责：

- 根据订单、滑点、费用和交易规则生成执行结果。
- 配合尚书省完成买卖操作。

### `xing_bu_justice.py`

刑部：风控和违规记录。

职责：

- 记录风控驳回、违规、熔断等事件。
- 为报告和告警提供风险事件来源。

### `gong_bu_works.py`

工部：行情数据清洗和指标前处理。

职责：

- 清洗 K 线数据。
- 统一字段、排序和基础质量检查。
- 为策略和回测提供干净行情输入。

## `src/strategies`：策略层

策略层定义策略接口、内置策略、策略库元数据和策略创建方式。

### 核心文件

- `base_strategy.py`：基础策略接口。
- `implemented_strategies.py`：内置策略实现，包含 `BaseImplementedStrategy` 和多套策略类。
- `strategy_factory.py`：策略工厂，根据配置和策略库创建运行时策略实例。
- `strategy_manager_repo.py`：策略管理仓库，负责策略元数据、启停、导入、更新、生成策略代码等。
- `user_strategy_template.py`：用户自定义策略模板。
- `strategy_01.py`：早期或单独策略示例。

### 策略运行契约

策略通常需要：

- 继承 `BaseStrategy` 或 `BaseImplementedStrategy`。
- 实现 `on_bar(kline)`。
- 返回标准信号字典，包含策略 ID、股票代码、方向、价格、数量等信息。
- 由 `ZhongshuSheng` 统一调度，而不是直接调用执行或资金模块。

推荐依赖方向：

```text
strategies -> utils.indicators / utils.runtime_params
strategies 不应直接依赖 server.py
strategies 不应直接操作数据库或 WebSocket
```

## `src/utils`：基础设施与横向工具

`utils` 是横向基础设施层，提供配置、数据源、指标、同步和通知能力。这里的代码应尽量可复用，不绑定具体 API 路由。

### 配置与运行参数

- `config_loader.py`：读取、合并、保存 `config.json` 和私有配置。
- `runtime_params.py`：运行时参数读取。
- `constants.py`：常量定义。
- `dependency_bootstrap.py`：启动前依赖检查。
- `backtest_baseline.py`：全局回测基线模板应用。

### 数据源 Provider

统一输出字段：

```text
code, dt, open, high, low, close, vol, amount
```

主要 provider：

- `data_provider.py`：默认 HTTP API 数据源。
- `tushare_provider.py`：TuShare 数据源。
- `akshare_provider.py`：AkShare 数据源。
- `mysql_provider.py`：MySQL 行情库。
- `postgres_provider.py`：PostgreSQL 行情库。
- `duckdb_provider.py`：DuckDB 本地行情库。
- `tdx_provider.py`：TDX / 通达信数据源。
- `data_factory.py`：统一构建 provider 的轻量工厂。

调用位置：

- `BacktestCabinet` 按 `data_provider.source` 构建回测数据源。
- `LiveCabinet` 按 `provider_type` 构建实盘数据源。
- `HistoryDiffSyncService` 用 provider 读取行情并写入目标库。
- `server.py` 的配置中心测试接口会临时构建 provider 做连通性检查。

### 数据同步与股票池

- `history_sync_service.py`：历史增量同步服务。
- `stock_list_refresh.py`：刷新股票池，支持 AkShare / TuShare / auto。
- `stock_manager.py`：股票列表加载和搜索。

历史同步要区分两个方向：

```text
data_provider.source              读取行情源
history_sync.direct_db_source     direct_db 模式下的写入目标库
```

例如：`source=tdx` 且 `direct_db_source=duckdb` 表示从 TDX 读行情，写入 DuckDB。

### 指标、筛选、通知

- `indicators.py`：均线、RSI、重采样等指标工具。
- `screener_data_provider.py`：选股器数据提供和过滤。
- `blk_loader.py`：TDX BLK 板块文件解析。
- `webhook_notifier.py`：Webhook 通知和失败事件持久化。
- `data_generator.py`：Mock 数据生成器。

## `src/evolution`：策略进化子系统

策略进化是一个相对独立的扩展子系统，用于生成、评估、筛选和入库新策略。

### 分层结构

```text
src/evolution/core/          编排核心、事件总线、运行时、基因模型
src/evolution/agents/        Researcher / Trader / Critic / Analysis / Committer
src/evolution/adapters/      回测、LLM、策略库、基本面、选股器等适配器
src/evolution/memory/        策略记忆、分析记录、基因运行记录
src/evolution/llm/           LLM 客户端工厂
src/evolution/algorithms/    遗传算法
src/evolution/persistence/   checkpoint 和状态持久化
src/evolution/platform/      外部平台服务聚合
src/evolution/runner/        独立运行入口
```

### 核心调用关系

```text
EvolutionRuntimeManager
  -> EvolutionOrchestrator
      -> Researcher 生成候选策略
      -> Trader / BacktestAdapter 评估策略
      -> Critic 审核策略质量
      -> StrategyLibraryCommitter 入库策略
      -> EventBus 发布进度事件
```

`BacktestAdapter` 会复用 `BacktestCabinet`，因此策略进化不是另一套回测系统，而是在现有回测能力外层增加生成和评估循环。

## `src/consistency`：一致性子系统

一致性模块用于比较实盘和回测在数据、信号、交易、报告上的差异。

目录职责：

- `collectors/`：采集实盘快照。
- `storage/`：保存快照。
- `replay/`：构建和存储回放运行。
- `comparators/`：比较快照、回测和执行差异。
- `reporting/`：生成一致性报告并持久化。
- `adapters/`：对接回测报告等外部数据结构。

典型关系：

```text
LiveCabinet
  -> LiveSnapshotCollector
      -> LiveSnapshotStore
          -> ReplayBuilder / ConsistencyComparator
              -> ConsistencyReportBuilder / ConsistencyReportStore
```

## `src/strategy_intent`：策略意图层

策略意图层把人类描述、市场状态或选股器结果转换成结构化意图。

主要文件：

- `strategy_intent.py`：意图数据结构。
- `human_intent_parser.py`：解析人工输入。
- `market_intent_generator.py`：根据市场状态生成意图。
- `screener_parser.py`：解析选股器条件。
- `intent_engine.py`：统一编排入口。

该层通常被策略管理、策略生成或选股相关功能调用，不直接执行交易。

## `src/tdx`：通达信能力层

该包与 `utils/tdx_provider.py` 不同：

- `utils/tdx_provider.py` 是行情数据源 provider。
- `src/tdx/` 是通达信公式、终端和交易网关相关能力。

主要文件：

- `formula_compiler.py`：把 TDX 公式转换/编译为 Python 策略代码。
- `terminal_bridge.py`：TDX 终端桥接、行情订阅、模拟网关和券商网关骨架。

调用关系：

```text
server.py / API
  -> src.tdx.formula_compiler
  -> src.tdx.terminal_bridge
  -> strategies / strategy_manager_repo
```

## `scripts`：脚本层

脚本层用于运维、批量任务、数据准备和诊断，不作为主业务包被长期运行服务强依赖。

常见脚本：

- `update_history_sync_stock_list.py`：刷新历史同步股票池。
- `batch_backtest_runner.py`：批量回测任务生成、执行和评分。
- `diagnose_duckdb_history_sync.py`：诊断 DuckDB 历史同步判重查询。
- `e2e_seed_data.py`：E2E 或演示数据准备。
- `linux&macOS系统启动.sh` / `start.ps1`：平台启动脚本。

## 数据流说明

### 单票回测数据流

```text
前端 / run_backtest.py
  -> server.py 或 BacktestCabinet
  -> ConfigLoader 读取配置
  -> Provider 拉取 K 线
  -> GongBuWorks 清洗
  -> ZhongshuSheng 调用策略生成信号
  -> MenxiaSheng 风控审核
  -> ShangshuSheng 执行订单
  -> HuBuRevenue 更新资金
  -> XingBuJustice 记录风险事件
  -> LiBuRites 生成报告和排行
  -> server.py 返回前端 / run_backtest.py 保存报告
```

### 实盘监控数据流

```text
run_live.py / 前端 start_live
  -> LiveCabinet
  -> Provider 获取最新行情
  -> 三省六部决策和执行链路
  -> LiveSnapshotCollector 采集快照
  -> WebhookNotifier / WebSocket 推送事件
  -> 前端展示状态、交易、告警和资金
```

### 历史同步数据流

```text
server.py / scripts/update_history_sync_stock_list.py
  -> stock_list_refresh 生成股票池

server.py / HistoryDiffSyncService
  -> 按 history_sync 配置确定时间范围和股票池
  -> 按 data_provider.source 构建读取 provider
  -> 读取 1min/多周期/日线行情
  -> 按 write_mode 和 direct_db_source 构建写入目标
  -> upsert 到 MySQL / PostgreSQL / DuckDB 或写入默认 API
  -> 记录 checkpoint 和同步记录
```

### 策略进化数据流

```text
前端 Evolution API
  -> EvolutionRuntimeManager
  -> EvolutionOrchestrator
  -> Researcher 生成候选策略
  -> BacktestAdapter 调用 BacktestCabinet 评估
  -> Critic 审核
  -> Memory / Store 保存过程
  -> StrategyLibraryCommitter 写入策略库
  -> EventBus 推送进度
```

## 分包依赖边界

推荐依赖方向：

```text
入口层(server/run_*) -> core -> ministries / strategies -> utils
evolution -> adapters -> core/strategies/utils
consistency -> core 输出的快照/报告结构
tdx -> strategies/utils 或 server API
```

避免的依赖方向：

- `utils` 不应反向依赖 `server.py`。
- `strategies` 不应直接依赖 `server.py`、WebSocket 或数据库连接。
- `ministries` 不应读取 HTTP 请求对象。
- `core` 不应写死前端细节。
- `server.py` 不应承载大量可复用业务算法。
- `evolution` 生成策略时不应绕过 `strategy_manager_repo` 直接散落写文件，除非是明确的临时输出。

## 新功能放置建议

### 新增数据源

放置位置：

- provider 实现：`src/utils/<name>_provider.py`
- 构建逻辑：`BacktestCabinet._build_provider()`、`LiveCabinet` 初始化分支、`server.py` 配置测试构建分支、`HistoryDiffSyncService` 运行时构建分支。
- 文档：`docs/data/数据源配置指南.md`

要求：

- 输出统一字段：`code, dt, open, high, low, close, vol, amount`。
- 提供 `check_connectivity()` 更利于配置中心检查。
- 支持 `fetch_minute_data()`，如有多周期则支持 `fetch_kline_data()`。

### 新增策略

放置位置：

- 手写内置策略：`src/strategies/implemented_strategies.py` 或独立策略文件。
- 策略元数据和入库：`strategy_manager_repo.py`。
- 创建入口：`strategy_factory.py`。

要求：

- 继承统一策略基类。
- 不直接下单，不直接改资金。
- 只产出信号，由三省六部链路执行。

### 新增回测规则

放置位置：

- 数据取数和流程级规则：`BacktestCabinet`。
- 风控规则：`MenxiaSheng` 或 `XingBuJustice`。
- 执行撮合规则：`BingBuWar` / `ShangshuSheng`。
- 成本和资金规则：`HuBuRevenue`。
- 指标统计：`LiBuRites`。

### 新增前端 API

放置位置：

- 路由入口：`server.py`。
- 复杂业务逻辑：下沉到 `src/core`、`src/utils`、`src/evolution` 或 `src/consistency`。

要求：

- API 层只做请求/响应编排。
- 与配置相关的逻辑复用 `ConfigLoader`。
- 与数据源相关的逻辑复用 provider。

### 新增一致性检查

放置位置：

- 采集：`src/consistency/collectors`
- 存储：`src/consistency/storage`
- 对比：`src/consistency/comparators`
- 报告：`src/consistency/reporting`

## 当前结构注意点

- `server.py` 当前集成了大量 API 和辅助逻辑，是主要膨胀点；新增功能应优先抽到包内再由 API 调用。
- `main.py` 是简化入口，不能代表完整回测能力；真实产品路径优先看 `BacktestCabinet`。
- `config.json` 中部分默认路径是 Windows 风格，在 macOS/Linux 下需要通过配置中心或私有配置修正。
- `data/` 目录包含运行数据、报告、缓存和股票池，不适合作为代码模块依赖。
- `docs/superpowers/` 下多为设计/计划记录，可作为历史上下文，不等同于当前运行代码。
